# /datavidence-healthanalysis:onboard-data

> Operationalizes `.claude/policies/analysis/data-onboarding.md`. Apply
> `../_shared/health-principles.md` (claim-to-evidence, PENDING_LOCAL_DECISION, no invention).
> Output follows `../_shared/spec-artifact.md` (simple artifacts path): a `data/README.md` +
> `config.example.yml`; no `<spec-dir>/` needed.

## When to use / when NOT to use
- Use it when: the user wants to connect to a data source, copy a dataset, set up the data folder,
  record where the data lives, or onboard raw files before any analysis begins.
- Do NOT use it when: validating schema, types, or contractual constraints — that is
  `validate-data-contract`. Do NOT use it for analysis, phenotyping, or indicator construction;
  this skill only CONNECTS/COPIES and documents the raw inputs.

## What it does (in order, no skipping)

### Step 1 — Read the policy taxonomy
Read `.claude/policies/analysis/data-onboarding.md`. Identify the data-size category:
- **Small / reference:** can be committed → copy into `data/raw/` (immutable, never edit in place).
- **Heavy / private / external:** connect-not-copy → point `config.yml` at origin; `data/raw/`
  stays empty or contains only a placeholder.

If the category is not yet determinable (no size/sensitivity information from the user), emit a
`PENDING_LOCAL_DECISION: data-size-and-sensitivity` and ask the user before proceeding further.

### Step 2 — Determine access mode
Ask (or infer from context):
- **Location:** local path, network share, URL, database, or cloud bucket?
- **Sensitivity:** public/open, restricted/institutional, patient-identifiable?
- **Size:** rough row count or file size?

Any unmade decision becomes `PENDING_LOCAL_DECISION: <what>` in the output. Do not silently default.

### Step 3 — Execute the connect/copy
- **Copy-light path:** `mkdir -p data/raw/<dataset-name>` + copy or document the copy command;
  record the checksum (`md5sum` or `sha256sum`). Raw files are committed; they are never overwritten.
- **Connect-heavy path:** write `config.yml` (git-ignored) with the absolute path; write
  `config.example.yml` (committed) with a placeholder and a human-readable comment. Paths resolve
  via `here` (R) or `rio` I/O; never hardcode an absolute path in analysis code.

Always add (or confirm) these `.gitignore` lines:
```
data/raw/        # omit this line on copy-light path when raw IS committed
config.yml
```

### Step 3b — Classify per regenerables.md
After executing the connect/copy, classify the dataset and record it in the provenance entry:
- **INPUT (committed source/reference data):** data that arrives from the outside world and
  cannot be regenerated by the project (e.g., a raw registry extract, a reference code list).
  These belong in `data/raw/` and are committed (copy-light path) or registered as an
  external input (connect-heavy path).
- **INTERMEDIATE (git-ignored, rebuilt):** data reconstructible from a heavy or external
  source by re-running a pipeline step (e.g., a derived analytic dataset, a cached API pull).
  These are git-ignored; include the rebuild command or pipeline reference in `sources.yml`.

Record the classification (`input` or `intermediate`) in `metadata/sources.yml` under a
`data_class` key. If the classification cannot be determined, emit
`PENDING_LOCAL_DECISION: data-class-input-vs-intermediate`.

### Step 4 — Write provenance
Populate (or append) `metadata/sources.yml` with one entry per raw input:
```yaml
- id: <dataset-id>
  source: "<origin URL, system name, or institution>"
  date_accessed: "<YYYY-MM-DD or PENDING>"
  access_mode: "<copy | connect>"
  checksum: "<md5/sha256 or PENDING>"
  notes: "<any access restrictions or re-request instructions>"
  data_class: "<input | intermediate>"  # per regenerables.md
```
If a field cannot be determined now, value is `PENDING_LOCAL_DECISION`.

### Step 5 — Emit the simple artifacts
Produce two files per `../_shared/spec-artifact.md` (simple artifacts):

**`data/README.md`** — one short section per dataset:
- What it is (one sentence), where it came from, access mode, date, immutability stance
  ("raw files in this folder are read-only inputs; never overwrite"), and .gitignore stance.
- Any open PENDINGs as a checklist at the bottom.

**`config.example.yml`** (committed, connect-heavy path only):
```yaml
# Copy this file to config.yml (git-ignored) and fill in your local paths.
paths:
  raw_data: "/FILL_IN/path/to/data/raw/<dataset-name>"   # absolute path on your machine
  # Add further path keys as needed; resolve in R with here::here() or rio::import().
```

No Quarto render needed for this skill — the artifact is the two files above plus the
`metadata/sources.yml` entry.

## Rules / invariants
- **Raw is immutable.** Copied data in `data/raw/` is never overwritten or edited in place.
- **No absolute paths in code.** All path resolution goes through `config.yml` + `here`/`rio`.
- **PENDING_LOCAL_DECISION, never silent defaults.** If the size/sensitivity/location is
  unknown, surface it; do not guess copy-vs-connect.
- **Record provenance.** Source, date, and checksum (or PENDING) in `metadata/sources.yml`.
- **Claim to evidence.** Every field in the provenance entry anchors to something the user
  provided or an observable fact; do not invent a source URL or checksum.
- **Scope boundary.** This skill stops at connection + minimal docs. Schema validation and
  quality checks belong to `validate-data-contract`, which reads what this skill registers.
